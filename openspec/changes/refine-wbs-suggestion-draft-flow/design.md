## Context

现有快速分解已具备规则生成能力，但“生成即入列表”的流程把建议草稿与正式分解项混在一起，用户无法在入列前做完整确认；同时提示语义混乱（信息与校验共用告警样式），容易造成“系统报错”误判。另一个核心问题是规则粒度过大，默认跨阶段生成任务，不符合“单需求点先在当前阶段拆解”的实际使用习惯。

## Goals / Non-Goals

**Goals:**
- 将建议生成流程改为“草稿确认后入列”，让用户拥有可控的确认节点。
- 标准规则改为阶段感知，按当前阶段生成建议条目，降低过度展开。
- 将紧前/紧后依赖关系前置到草稿阶段，确保关联性不是后补动作。
- 手动表单默认折叠，规则生成为主路径、手动补录为按需路径。
- 统一提示分级（信息/警告/错误）并支持行级定位。

**Non-Goals:**
- 不重写后端核心 WBS 模型。
- 不引入 AI 推理能力。
- 不覆盖其他模块表单页面，仅聚焦快速分解弹窗。

## Decisions

### 1) 双区模型：草稿区 + 已确认分解列表

决策：新增 `draftRows` 作为建议草稿容器；`wbsRows` 保留为已确认列表。只有“批量加入分解列表”后才进入 `wbsRows`。

原因：草稿与正式数据语义分离，降低误操作与误解。

备选：继续直接入列后再编辑，用户心智负担高。

### 2) 阶段感知的标准规则

决策：规则生成接口增加 `mode` 与 `targetStage` 输入。`mode=standard` 时默认生成当前阶段 2-4 条；`mode=complete` 才跨阶段展开。

原因：符合用户对“单需求点”的实际粒度预期。

备选：维持一刀切跨阶段模板，继续产生过度拆分。

### 3) 依赖交互前置到草稿阶段

决策：草稿行可直接配置紧前任务（名称多选），紧后任务实时反推只读展示；加入列表前执行依赖合法性校验。

原因：把“任务关联性”变成确认入口的硬约束，避免后置修补。

### 4) 手动表单按需展开

决策：默认隐藏完整手动字段区，提供“展开手动填写/收起”开关。

原因：提升默认清爽度与主路径效率，同时保留专业用户精细录入能力。

### 5) 提示分级

决策：
- 信息态：建议生成成功、草稿未入列状态
- 警告态：草稿存在可修正问题
- 错误态：阻断提交或加入列表

原因：避免“生成成功被渲染成校验错误”的语义冲突。

## Risks / Trade-offs

- [Risk] 增加草稿区后交互步骤变长 → Mitigation：支持批量加入与批量编辑，减少额外点击。
- [Risk] 草稿与列表双状态同步复杂 → Mitigation：明确单向流转（草稿 -> 列表），避免隐式双向绑定。
- [Risk] 依赖选择源混合（草稿+已确认）可能引起 ID 混淆 → Mitigation：草稿临时标识与正式 ID 分层管理。
- [Risk] 阶段感知规则命中不准 → Mitigation：允许用户切换档位并重生成。

## Migration Plan

1. 扩展规则建议接口支持 `mode/targetStage`。
2. 前端引入草稿区状态与“批量加入”动作，替换当前直接入列逻辑。
3. 在草稿区接入紧前选择、紧后反推和加入前校验。
4. 调整提示组件与文案分级样式。
5. 手动表单改为折叠区并联调。

回滚策略：关闭新流程开关，恢复“生成后直接入列”的旧行为。

## Open Questions

- 草稿依赖引用草稿任务时，是否允许临时 ID 参与校验。
- “标准档位”默认生成条数是否按项目类型可配置。
- 批量加入时，是否支持“仅加入已勾选且通过校验行”。

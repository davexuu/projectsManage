## Context

当前系统在 WBS 任务分解录入场景存在三个主要问题：

1. 录入流程以“字段堆叠表单”为主，缺少按项目上下文组织的交互，开发者需要频繁切换思路并重复输入信息。
2. 新增记录字段未区分核心字段与辅助字段，导致录入负担较重，且部分字段可以由系统默认值、项目上下文或后续编辑补充。
3. 表格样式缺乏统一规范，行高、边框、表头、状态展示与空态表现不一致，影响信息扫描效率。

本次变更面向前端交互和展示层规范，目标是在不改变核心业务语义的前提下，提升 WBS 录入效率与表格使用体验，并为后续页面统一样式提供基础。

约束与前提：
- 需要兼容现有 WBS 数据结构与接口，优先通过前端字段映射、默认值和校验层优化实现。
- “非必要字段”的最终名单与填写文案需要业务确认，但设计需预留可配置能力。
- 当前 proposal 中已定义 3 个能力：`wbs-entry-flow`、`wbs-form-guidance`、`table-ui-standard`，设计应覆盖这三类能力。

## Goals / Non-Goals

**Goals:**
- 将 WBS 新增/编辑交互调整为“项目上下文驱动 + 分组/分步填写”的流程，降低重复输入与理解成本。
- 明确字段优先级（必填/建议填/可选/自动生成），在 UI 中提供统一填写要求提示样式。
- 建立统一表格样式标准，使 WBS 列表和其他业务列表具备接近 ElementUI Table 的视觉与交互体验。如果有现成的UI组建，优先使用组件。
- 通过可配置字段定义（标签、是否必填、提示文案、示例、校验规则）减少后续改动成本。

**Non-Goals:**
- 不在本次设计中重构后端接口或数据库模型（除非出现无法兼容的字段约束）。
- 不覆盖所有历史页面的表格改造实现，仅先定义标准并落地到 WBS 相关页面。

## Decisions

### 1) WBS 录入交互采用“基础信息 + 分解项明细”分段布局（可选步骤式）

决策：
- 将录入表单拆为两个层次：
  - 基础上下文区：项目、迭代/版本、负责人（如适用）、时间范围等
  - WBS 分解项区：任务项列表/行内录入（角色/人员、任务内容、工时等）
- 若当前页面已有弹窗表单，优先使用“分组区块 + 吸顶小标题”；仅在字段较多时升级为步骤式（Step 1 基础信息 / Step 2 分解项 / Step 3 确认提交）。

原因：
- 用户反馈的核心是“逻辑不对”和“填写不便捷”，问题更偏信息组织而非单字段控件。
- 分段布局比一次性大表单更易理解，也能复用已有页面结构，改造成本低于完整向导式。

备选方案与取舍：
- 方案 A：保持单表单，仅优化字段顺序。成本低，但无法根治认知负担。
- 方案 B：完整多步骤向导。引导性强，但对快速录入场景操作路径更长。
- 采用折中方案：默认分组，复杂场景可开启步骤式。

### 2) 字段模型引入“显示层配置”以支持精简与填写要求标注

决策：
- 为新增/编辑表单建立字段配置结构（可在前端常量或页面配置中实现），至少包含：
  - `label`
  - `required`
  - `priority`（core/recommended/optional）
  - `placeholder`
  - `helpText`
  - `example`
  - `validator`
  - `visibleWhen`（按项目类型或上下文联动显示）
  - `defaultValueResolver`（从项目上下文/用户信息推导默认值）
- UI 展示规则：
  - 必填字段显示 `*` 与简短填写要求
  - 推荐字段显示辅助说明
  - 可选字段默认折叠到“更多字段”

原因：
- 你的需求同时包含“有些字段可以不用要”和“把填写要求标注好”，两者本质上都需要统一字段元信息驱动。
- 配置化字段层可以减少散落在模板代码中的条件判断与文案硬编码。

备选方案与取舍：
- 方案 A：在表单模板中手写每个字段说明与显隐逻辑。短期快，但后续维护成本高。
- 方案 B：抽象成动态表单引擎。灵活性高，但本次范围过大。
- 采用轻量配置化方案，满足当前需求并保留演进空间。

### 3) WBS 分解项录入支持“批量行编辑”与常用值继承

决策：
- 在分解项区域使用表格化录入（editable rows），支持新增行、复制行、删除行。
- 对可继承字段（如项目、版本、负责人、日期范围、默认工时单位）提供行级默认继承，减少重复录入。
- 对高频字段提供快捷输入策略（如 Enter 新增下一行、复制上一行部分值）。

原因：
- WBS 分解项天然是多条记录录入场景，逐条弹窗新增会显著降低效率。
- 批量行编辑符合开发者“连续输入”习惯，能直接回应“便捷填写”的诉求。

备选方案与取舍：
- 方案 A：保留单条新增弹窗。实现简单，但效率低。
- 方案 B：纯表格行内编辑。高效，但校验与错误提示复杂。
- 采用混合策略：主流程行内编辑，复杂字段可下钻弹层编辑（如需要）。

### 4) 表格视觉标准采用“Element 风格兼容层”而非直接依赖 ElementUI

决策：
- 抽象一个项目内统一表格样式层（例如表格容器类 + 通用单元格/表头样式 token），实现接近 ElementUI Table 的体验：
  - 清晰表头背景与文字层级
  - 边框/分割线规范
  - 斑马纹（可配置）
  - hover 高亮
  - 行高/内边距统一
  - 空状态、加载态、固定列阴影风格统一
- 若项目当前已有表格组件封装，则在封装层扩展 `variant="element-like"` 或默认主题；若没有，则先为 WBS 页面落地并沉淀样式变量。

原因：
- 用户明确希望“类似 ElementUI table”的样式，但直接引入/切换 UI 框架会扩大影响面。
- 样式兼容层更可控，能在现有技术栈内逐步推广。

备选方案与取舍：
- 方案 A：直接引入 ElementUI/Element Plus 表格。视觉成熟，但技术栈兼容与迁移成本高。
- 方案 B：页面局部 CSS 修补。见效快，但缺乏统一标准。
- 采用通用样式层方案，兼顾一致性与落地成本。

### 5) 校验与提示策略分层：提交前聚合 + 字段级即时反馈

决策：
- 字段级：失焦或输入后展示格式/范围校验提示。
- 行级（分解项）：标记具体行和列，避免仅在顶部报错。
- 提交级：聚合显示错误摘要，并支持定位到对应字段/行。
- 填写要求文案与校验报错文案分离：前者做指导，后者说明错误原因。

原因：
- 批量录入场景下，仅靠提交时报错会导致定位困难，降低可用性。
- “填写要求标注”若与报错混写，容易造成文案冗长和认知噪音。

备选方案与取舍：
- 方案 A：仅提交时校验。实现简单，但体验差。
- 方案 B：全部实时校验。反馈及时，但噪音大。
- 采用分层校验策略以平衡效率与打扰程度。

## Risks / Trade-offs

- [风险] 字段精简后可能遗漏少数业务场景需要的信息 → Mitigation：将非核心字段收纳到“更多字段”，先隐藏不删除，并记录使用情况。
- [风险] 行内编辑表格会增加表单状态管理复杂度 → Mitigation：采用统一数据模型（rows[]）和字段配置驱动校验，避免多处状态源。
- [风险] Element 风格表格与现有页面样式冲突 → Mitigation：通过 scoped 类名/主题变量隔离，先在 WBS 页面灰度落地。
- [风险] 填写要求文案过多导致视觉拥挤 → Mitigation：默认展示简版说明，详细示例通过 tooltip/帮助图标展开。
- [风险] 业务方对“非必要字段”定义不一致 → Mitigation：在设计阶段列出字段分级清单与确认项，待 specs 固化要求。

## Migration Plan

1. 梳理 WBS 新增/编辑现有字段清单，按“必填/建议/可选/自动推导”分类。
2. 落地字段配置结构与校验/说明渲染逻辑（先在 WBS 页面使用）。
3. 重构 WBS 录入页面布局为分段式，并引入分解项表格化录入区域。
4. 实现统一表格样式 token/类名，并在 WBS 列表与分解项表格中应用。
5. 联调并验证接口字段映射兼容性，补充缺省值处理。
6. 灰度发布（如支持），收集填写效率与错误率反馈，再决定是否推广到其他表格页面。

回滚策略：
- 保留旧录入入口或通过配置开关切回旧表单布局。
- 表格样式采用页面级类名开关，必要时回退为原样式。

## Open Questions

- 哪些“新增记录字段”可以明确降级为可选，哪些必须保留为必填？需要业务确认最终清单。
- WBS 分解项中的“人物分解项”字段是否存在固定角色枚举，还是支持自由输入？
- 是否需要导入/复制历史 WBS 模板（按项目或版本复制）以进一步提升录入效率？
- 当前项目是否已有统一表格组件封装，可直接承载 `element-like` 风格主题？
- 表格样式统一是仅限 WBS 页面，还是本次就同步覆盖其他高频列表页？

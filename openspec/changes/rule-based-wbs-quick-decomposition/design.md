## Context

当前项目已具备 WBS 批量录入、计划校验和甘特依赖展示能力，但“快速分解WBS”仍以字段手填为主，用户在不熟悉 PMP 术语时难以快速产出可执行计划。现有问题集中在四处：
- 输入门槛高：用户只能先想清楚阶段、工作包、任务细节再填表。
- 结构字段可用性差：前置任务仍依赖手填 ID，关联关系建立困难。
- 编码维护成本高：`wbsCode` 需要手动填写或补录，易重复或错序。
- 信息展示负担重：说明文案长期常显，影响快速录入效率。

该改造是跨前后端与元数据配置的联动变更，需要保持与现有 `/wbs/validate-plan`、`/wbs/batch` 兼容，并确保既有录入流程不被破坏。

## Goals / Non-Goals

**Goals:**
- 提供规则引擎驱动的“单句输入 -> 建议分解行”能力，降低快速分解的认知门槛。
- 提供自动 WBS 编码策略，默认由系统生成并稳定排序。
- 提供前置/后继任务可视化关联操作，替代手填任务 ID。
- 优化字段指导信息呈现方式（icon 悬浮），减少页面视觉噪音。
- 复用现有批量校验和批量提交链路，控制改造范围。

**Non-Goals:**
- 本次不引入大模型能力，不依赖外部 AI 服务。
- 本次不实现资源平衡、自动排程重算等高级计划算法。
- 本次不重构所有历史模块录入页面，仅覆盖计划编排工作台快速分解入口。

## Decisions

### 1) 采用“规则分类 + 模板生成”的确定性拆解引擎

决策：新增后端规则拆解服务，对“我要做什么”进行意图分类（新增/修复/优化/合规），再按模板输出建议分解行。

原因：
- 无需等待 AI 接口即可落地；结果可预测、可回归。
- 输出结构与现有 `wbs` 字段天然对齐，可直接进入批量校验与提交。

备选：
- 前端本地硬编码规则：实现快，但规则迭代与一致性较差。
- 直接接入 AI：短期不可行，且不可控性更高。

### 2) 自动编码由后端统一生成，前端只读展示

决策：`wbsCode` 生成逻辑放在服务端（批量/单条均适用），前端展示为只读或弱可见，不作为用户录入必填项。

原因：
- 避免并发写入导致重复编码与排序不一致。
- 统一编码规则可保证列表、树形和导出一致。

备选：
- 前端预生成编码再提交：会出现并发冲突和回填复杂度。

### 3) 依赖关系录入改为“名称选择 + ID 存储”

决策：在快速分解列表中提供“关联前置任务”选择器（可多选），UI 展示任务名称，提交时转换并持久化为 `predecessorTaskIds`。

原因：
- 贴合用户心智，显著降低输入错误率。
- 与现有 `validate-plan`、甘特依赖渲染模型兼容。

备选：
- 保留手填 ID 并增加说明：成本低但收益有限。

### 4) 字段指导信息采用“默认收敛 + 悬浮查看”

决策：关键字段说明改为 info icon + Tooltip/Popover；错误提示仍走校验消息，保持“指导提示”与“错误提示”分离。

原因：
- 提升信息密度控制，减少页面拥挤。
- 与既有字段元数据（helpText/example）兼容，不需要破坏 schema 结构。

## Risks / Trade-offs

- [Risk] 规则模板覆盖不足导致建议行质量不稳定 → Mitigation：首版仅覆盖四类高频意图并提供“人工编辑优先”机制。
- [Risk] 自动编码引入历史数据排序差异 → Mitigation：仅对新建记录自动编码；历史记录维持原值，不做强制重排。
- [Risk] 依赖选择器在大数据量下可用性下降 → Mitigation：增加按阶段/关键字过滤与分页搜索。
- [Risk] 前后端兼容窗口内存在双模式（手填/自动）行为差异 → Mitigation：通过 feature flag 灰度并补充回归用例。

## Migration Plan

1. 新增规则拆解接口（或服务方法）并输出标准建议行结构，不改动现有创建接口。
2. 在快速分解弹窗接入“我要做什么”输入与“生成建议”操作，建议行进入现有待提交列表。
3. 服务端实现自动编码逻辑，前端将 `wbsCode` 调整为只读展示。
4. 快速分解内新增前置任务可视关联组件，提交时转换为 `predecessorTaskIds`。
5. 字段指导文案改为 icon 悬浮显示，并验证错误提示链路不受影响。
6. 通过工作台流程回归后再默认开启 feature flag；保留回滚开关回到旧流程。

回滚策略：
- 关闭新 feature flag，恢复原有手工录入与字段展示。
- 保留新接口但不在前端调用，不影响既有 API 契约。

## Open Questions

- 自动编码层级规则是否需要显式支持“父子任务跨阶段”场景，还是先限制同阶段内编码。
- 规则模板是否支持按项目类型（研发/运维/合规）分组加载。
- 是否在本次同时提供“批内依赖关系校验”（同次生成任务互为依赖）增强。

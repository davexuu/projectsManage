## Context

当前任务看板位于 `frontend/src/features/kanban/KanbanBoard.tsx`，已存在基础拖拽交互（`draggable`、`onDrop`）和任务卡片信息展示逻辑。看板拖拽后通过 `api.update('wbs', id, payload)` 调用后端 `PUT /wbs/:id` 更新任务。

后端 `PUT /wbs/:id` 使用 `createWbsSchema` 做全量校验，要求提交完整 WBS 字段（如 `taskDetail`、`deliverable`、`isCritical` 等）。看板场景如果使用列表行数据直接更新状态，容易因字段缺失、类型不匹配或脏字段导致校验失败，从而出现“拖拽无法变更状态”的问题。

卡片上阶段和计划起止时间虽有展示代码，但需要通过规格明确展示要求、空值处理和数据来源，避免字段缺失或未来重构导致展示回退。

## Goals / Non-Goals

**Goals:**
- 让用户可以在任务看板中通过拖拽任务卡片跨状态列变更任务状态，并完成后端持久化。
- 明确拖拽更新成功/失败时的前端行为（乐观更新、回滚、提示）。
- 明确任务卡片必须展示阶段与计划起止时间，以及缺失值的显示规则。
- 保持现有项目权限控制与 WBS 数据模型兼容，不破坏现有编辑页面的全量更新流程。

**Non-Goals:**
- 不重做看板整体 UI 结构或样式体系。
- 不引入新的拖拽库（优先基于现有 HTML5 拖拽实现修复）。
- 不修改 WBS 业务字段定义（如阶段枚举、状态枚举）。
- 不实现批量拖拽、排序持久化或跨项目拖拽。

## Decisions

### 1) 新增 WBS 状态专用更新接口（推荐 `PATCH /wbs/:id/status`）

决策：
- 为看板拖拽提供一个仅更新状态的后端接口，输入最小化为 `projectId` 和 `currentStatus`（必要时可包含审计字段，当前不扩展）。
- 复用现有鉴权/项目权限校验模式，避免看板使用全量 `PUT /wbs/:id`。

原因：
- 当前 `PUT` 接口依赖 `createWbsSchema` 全量字段，和看板拖拽“只改状态”的需求不匹配。
- 专用接口可降低前端拼装 payload 的复杂度，减少拖拽失败概率。
- 后续若补充状态流转规则，也更适合在专用接口中集中处理。

备选方案（未选）：
- 继续复用 `PUT /wbs/:id`，前端先获取完整数据再回传全量字段。问题是耦合高、易受字段变更影响，且看板交互延迟更大。
- 放宽 `createWbsSchema` 为部分更新。问题是会影响通用 WBS 编辑语义，风险更大。

### 2) 前端采用乐观更新 + 失败回滚

决策：
- 拖拽落列后先在前端更新卡片状态，随后调用状态专用接口持久化。
- 接口失败时回滚到拖拽前状态，并给出错误提示。

原因：
- 提升拖拽交互即时反馈，避免等待接口完成才移动卡片。
- 与当前看板实现方向一致，改造成本低。

备选方案（未选）：
- 等接口成功后再移动卡片。实现简单，但交互迟滞明显。

### 3) 明确卡片字段展示规则（阶段 + 起止时间）

决策：
- 卡片必须显示 `level1Stage` 和计划起止时间（来自 `plannedStartDate` / `plannedEndDate`）。
- 时间展示采用统一格式（如 `MM-DD`），缺失值展示“待定”，两端都缺失时显示“计划：待定”。

原因：
- 需求目标是提升看板视图中的任务上下文信息密度。
- 通过规格固定展示规则，避免后续实现出现“有时显示、有时不显示”的不一致。

备选方案（未选）：
- 仅在 hover 或详情中展示阶段/日期。不能满足看板快速识别需求。

### 4) 数据刷新策略：单卡更新优先，全量刷新作为兜底

决策：
- 正常拖拽仅更新本地状态，不立即全量刷新。
- 当接口返回与本地状态不一致或服务端补充字段变化时，可在成功后触发一次局部/全量刷新（作为后续优化，不强制首版实现）。

原因：
- 降低交互延迟与请求量。
- 给后续一致性优化保留空间。

## Risks / Trade-offs

- [风险] 后端新增 `PATCH` 接口后，前端仍误用全量 `PUT` 导致行为不一致
  → Mitigation：在看板中单独封装 `api.updateWbsStatus`，避免复用通用 `update`。

- [风险] 状态流转规则（如禁止直接跳转）若后续加入，可能导致拖拽交互与业务规则冲突
  → Mitigation：在 spec 中先定义允许拖拽后的错误反馈；接口返回业务错误时前端必须回滚并提示。

- [风险] 看板列表数据中个别任务缺少计划日期，显示看起来像“未生效”
  → Mitigation：明确空值显示规则（“待定”），并在卡片上固定占位文案。

- [风险] HTML5 拖拽在不同浏览器/容器环境行为差异
  → Mitigation：保留当前实现并增加最小交互测试清单；必要时后续再升级拖拽库。

## Migration Plan

- 本次变更以增量方式进行，不涉及数据库迁移。
- 新增接口不替换现有 `PUT /wbs/:id`，避免影响现有编辑页面。
- 若前端看板切换到状态专用接口后出现异常，可临时回退到原逻辑（但保留卡片展示改动）。

## Open Questions

- 是否需要对 WBS 状态更新引入与变更单相同的状态流转校验（例如限制“已完成”回退）？若当前无明确业务规则，可先沿用现状允许任意目标状态。
- 看板卡片是否需要同时展示“实际起止时间”或仅展示“计划起止时间”？当前 proposal 范围按计划时间处理。

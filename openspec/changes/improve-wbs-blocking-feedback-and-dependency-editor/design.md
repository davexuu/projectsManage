## Context

当前 WBS 快速分解在 `DynamicForm` 中承担了建议生成、草稿确认、依赖编辑、批量提交等多类交互。最近迭代把“生成即入列”改成“草稿确认后入列”，但仍有三个落差：  
1) “当前阶段”选择与后端标准档回退策略叠加后，用户难以判断该字段是否有意义；  
2) 草稿表格内直接放置多选依赖控件，在窄列和滚动表格下可用性差；  
3) 后端 409 阻断（未审批变更影响基线）仅以顶部提示展示，缺少明确操作指引。  

本次设计目标是让用户在不理解内部规则的情况下，也能快速知道“出了什么问题、怎么改、去哪处理”。

## Goals / Non-Goals

**Goals:**
- 去掉快速建议区“当前阶段”手动选择，降低认知负担。
- 将依赖编辑改为“行内按钮 -> 弹窗”，提升可操作性与可读性。
- 对阻断类错误启用弹窗级提示，文案直白且提供可操作动作。
- 保持现有后端规则约束不变，仅优化前端交互与提示路径。

**Non-Goals:**
- 不修改 WBS/变更单领域规则（例如 409 阻断规则本身）。
- 不引入 AI 能力或新增后端复杂策略引擎。
- 不重构非 WBS 模块提示体系。

## Decisions

### 1) 移除“当前阶段”手动选择，改为系统策略说明

决策：快速建议区仅保留“建议档位 + 一句话描述”，不再显示“当前阶段”下拉。  
策略说明文案明确：  
- 轻量：优先最少建议；  
- 标准：优先当前阶段模板，不足时补充邻近阶段；  
- 完整：可跨阶段展开。

原因：当前阶段不是强约束，保留控件会制造“我选了为什么没完全按这个阶段”的误解。  
备选：保留控件并增加解释 tooltip；仍需要用户理解回退逻辑，负担较高。

### 2) 依赖编辑从单元格多选切换为弹窗编辑

决策：草稿表每行提供“编辑依赖”按钮，打开弹窗统一管理该行紧前任务；弹窗内提供清晰的候选任务列表、搜索和已选预览；保存后回写草稿行。

原因：表格内多选在横向滚动、窄列、分页下易错且难读。弹窗可容纳更完整信息（任务名、阶段、工作包、时间）。
备选：继续单元格多选并加宽列；会挤压其他核心列，且移动端/小屏体验仍差。

### 3) 阻断错误使用强提示弹窗并给出“怎么改”

决策：对后端 409 且命中“存在未审批且影响里程碑/WBS的变更”时，统一弹出 `Modal.error`（或等价强提示），内容包含：
- 问题：当前存在未审批变更（展示变更编号）
- 影响：暂时不能提交 WBS/里程碑基线修改
- 操作：去“变更管理”审批/驳回后再提交
- 按钮：`去处理变更单`、`我知道了`

原因：顶部 Alert 容易被忽略，且不告诉用户下一步。阻断类问题必须显式打断并提供行动入口。  
备选：仅改 Alert 样式加粗；仍不够强、操作链路不闭环。

### 4) 提示分层规则固化

决策：
- 信息：生成成功、草稿加入成功（轻提示）
- 警告：可修复冲突（行级问题）
- 错误：阻断提交（弹窗 + 行动入口）

原因：让反馈强度与业务风险一致，避免“所有提示都在一层”。

## Risks / Trade-offs

- [Risk] 移除当前阶段后，部分高级用户会担心失去控制  
  → Mitigation：在标准档文案中明确“优先当前阶段，不足补邻近阶段”，并保留完整档可选。
- [Risk] 弹窗编辑依赖增加一步点击  
  → Mitigation：支持快速入口（行内按钮）和弹窗内一次性保存，减少反复开关。
- [Risk] 阻断弹窗引导到变更管理可能跨页面上下文丢失  
  → Mitigation：保留当前草稿数据，处理后返回可继续提交。

## Migration Plan

1. 调整快速建议区 UI，移除“当前阶段”控件与前端传参。
2. 实现草稿行“编辑依赖”按钮和依赖编辑弹窗，替换单元格内多选。
3. 统一提交错误处理：识别 409 阻断并弹窗化，加入“去处理变更单”动作。
4. 联调验证：生成建议 -> 编辑依赖 -> 批量加入 -> 提交 -> 阻断处理 -> 回提。

回滚策略：保留旧交互开关；若新交互异常，可暂时回退到现有单元格编辑与顶部提示模式。

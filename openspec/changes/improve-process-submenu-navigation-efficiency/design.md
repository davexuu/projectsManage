## Context

当前过程工作区在 `ProcessWorkspace` 中以按钮集合展示子菜单，入口数量增长后定位成本明显上升，且缺乏稳定的层级结构与当前态反馈。现有路由已按过程分段（如 `/process/start`、`/process/plan`），具备承载结构化导航的信息基础，但导航组件与路由状态同步机制尚未统一。目标是在不改变既有业务页面语义的前提下，提升过程内切换效率并保证移动端可用性。

## Goals / Non-Goals

**Goals:**
- 建立统一的过程子菜单信息模型，覆盖启动/规划/执行/监控/收尾五大过程。
- 提供高效导航交互：当前项高亮、分组展示、最近访问快捷入口。
- 统一导航状态与路由状态，确保刷新、返回、深链进入时导航状态正确。
- 在桌面端与移动端提供不同呈现策略，但保证相同的信息架构。
- 提供可验证的导航效率指标（点击步数、到达时间、误跳转率）。

**Non-Goals:**
- 不在本次变更中重构各业务模块页面（WBS、风险、周报等）的内部交互。
- 不改动后端 API、权限模型和数据库结构。
- 不引入新的全局状态库，仅使用当前 React 状态管理能力完成导航状态同步。

## Decisions

### Decision 1: 引入过程导航配置中心，替代按钮硬编码
- 方案：在前端新增过程导航配置（过程 -> 分组 -> 子菜单项 -> 路由），由 `ProcessWorkspace` 与侧边/页内导航组件共用。
- 原因：避免入口定义分散在多个组件，降低后续新增子菜单的改动成本。
- 备选方案：继续在 `ProcessWorkspace` 中按过程写按钮数组；缺点是结构可读性差、扩展后维护成本高。

### Decision 2: 导航形态采用“桌面分组导航 + 移动抽屉导航”
- 方案：桌面端展示可折叠分组导航，移动端使用抽屉或分段菜单承载同一数据源。
- 原因：在有限空间中保留层级信息，减少横向拥挤和误触。
- 备选方案：统一使用按钮网格；缺点是入口多时滚动长、难体现层级关系。

### Decision 3: 路由作为唯一真源，导航状态派生
- 方案：以 `location.pathname + query` 推导当前激活菜单；最近访问记录写入 `sessionStorage`。
- 原因：保证浏览器前进/后退、深链访问时状态一致，避免双向同步冲突。
- 备选方案：导航维护独立 active state 再推路由；缺点是容易出现路由与导航高亮不一致。

### Decision 4: 以轻量埋点/日志评估导航效率
- 方案：前端记录导航点击和页面到达事件（本地埋点接口或控制台统计钩子），用于对比改造前后关键指标。
- 原因：效率改造需要可量化验收，而不仅是主观体验。
- 备选方案：仅人工体验验收；缺点是不可重复、不可持续追踪。

## Risks / Trade-offs

- [风险] 导航结构调整可能影响老用户操作习惯。 → Mitigation: 保留核心入口命名不变，提供短期引导文案与兼容跳转。
- [风险] 路由映射遗漏会导致“有页面无入口”或“有入口空白”。 → Mitigation: 建立导航配置校验清单，并做过程全覆盖回归。
- [风险] 移动端抽屉交互层级增加可能影响首屏效率。 → Mitigation: 将“最近访问”置顶，降低常用路径点击层级。
- [权衡] 引入配置中心会增加一次性重构成本。 → Mitigation: 后续新增/调整子菜单成本显著降低。

## Migration Plan

1. 增加过程导航配置与类型定义，映射现有过程子菜单。
2. 在 `ProcessWorkspace` 接入新导航组件，并保留旧按钮实现作为短期回退开关（仅开发期）。
3. 对 `App.tsx` 中过程路由入口进行导航状态对齐，修正激活态判定。
4. 增加最近访问记录逻辑与基础埋点钩子。
5. 在桌面端/移动端分别完成交互与回归验证，确认指标达到验收阈值。
6. 移除旧按钮式过程子菜单实现与临时开关。

回滚策略：保留旧导航组件实现分支；若发布后出现严重可用性问题，快速切回按钮导航渲染路径。

## Open Questions

- 最近访问数量上限是否固定为 3 或 5，是否需要按角色区分？
- 移动端过程导航是全局抽屉还是仅过程页局部抽屉，是否与现有 Sider 行为冲突？
- 导航效率指标最终由前端埋点还是由测试脚本采集作为准入标准？

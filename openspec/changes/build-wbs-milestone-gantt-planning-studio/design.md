## Context

当前系统已具备 WBS、里程碑、甘特图三类能力，但分别分布在模块化 CRUD 页与图表页，缺少统一的计划编排入口。前端已有 WBS 批量录入雏形（行级新增/复制/批量提交），后端仍以单条 WBS 创建接口为主，数据模型缺少 WBS 树结构和依赖关系字段，导致无法稳定支持关键路径、依赖冲突校验和任务-里程碑强关联。

约束：
- 现有模块化表单/表格体系需要保持兼容，避免一次性重写所有模块。
- 现有 `wbs`、`milestones` API 已被多个页面依赖，需要向后兼容。
- 需要先满足“快速录入 + 直观展示”，再逐步增强自动排程能力。

## Goals / Non-Goals

**Goals:**
- 提供项目级“计划编排工作台”，在同一入口完成 WBS 快速录入、里程碑编排、甘特图查看与问题定位。
- 建立可扩展的 WBS 结构化模型（层级、依赖、里程碑关联），支撑一致性校验。
- 保留并增强现有批量录入效率能力（上下文固定、继承、复制、批量校验）。
- 建立列表与图表双向定位，降低计划维护成本。

**Non-Goals:**
- 不在本次实现完整自动排程引擎（资源平衡、自动重排）。
- 不强制改造所有历史模块页面，仅聚焦计划编排主链路。
- 不引入新的重型 UI 框架替换当前 Ant Design 技术栈。

## Decisions

### 1) 新增“计划编排工作台”作为统一入口

决策：在前端新增独立页面（如 `PlanningStudio`），包含三块区域：
- WBS 树 + 明细批量编辑
- 里程碑时间线/列表
- 甘特图（任务条 + 里程碑）

原因：
- 现有入口分散，跨页操作成本高。
- 统一入口能建立一致筛选上下文（项目、阶段、时间窗、负责人）。

备选：
- 继续沿用多页面跳转并通过 query 传参。实现快但心智负担高。
- 采用新工作台，改造成本可控且收益明确。

### 2) WBS 数据模型扩展为“向后兼容的结构化字段”

决策：在 `WbsTask` 增加可空字段：
- `wbsCode`（层级编码，例 `1.2.3`）
- `parentTaskId`（父任务）
- `predecessorTaskIds`（依赖任务 ID 列表，JSON 或关联表）
- `milestoneId`（关联里程碑）
- `sortOrder`（同层排序）

原因：
- 关键路径、依赖校验、树形展示都需要稳定结构字段。
- 可空扩展能保持旧数据与旧接口兼容。

备选：
- 仅前端维护临时树结构，不改后端。短期可行但一致性不可控。
- 采用后端结构化字段，作为长期基线。

### 3) API 采用“兼容旧接口 + 新增批量与校验接口”

决策：
- 保留现有单条 `POST /wbs`。
- 新增 `POST /wbs/batch` 批量写入与事务回滚。
- 新增计划校验接口（如 `POST /wbs/validate-plan`），返回行级/字段级冲突信息。
- 里程碑接口增加与 WBS 关联查询能力。

原因：
- 批量录入场景需要原子提交与统一校验结果。
- 保留旧接口避免影响其他既有页面。

备选：
- 仅前端循环调用单条创建。简单但失败回滚和性能较差。
- 新增批量接口，提升一致性与体验。

### 4) 甘特图保持现有库，增强数据映射与联动能力

决策：继续使用 `frappe-gantt`，在数据层补充：
- 依赖线映射
- 里程碑样式与阶段泳道过滤
- 点击图元反向定位到编辑表格行

原因：
- 现有实现已可运行，迁移图表库成本高。
- 当前缺陷在数据语义而非图表基础能力。

备选：
- 更换甘特图库。能力更全但代价高。
- 保持库不变，先补齐模型与交互。

### 5) 校验体系分层：输入校验 + 计划一致性校验

决策：
- 输入校验：字段必填、日期格式、开始/结束先后关系。
- 一致性校验：依赖环、依赖任务日期矛盾、里程碑无支撑任务、跨阶段异常。
- UI 呈现：提交前摘要 + 行级定位 + 图表标色。

原因：
- 批量计划录入最怕“校验失败后难定位”。
- 一致性校验是计划可视化质量的基础。

## Risks / Trade-offs

- [风险] 数据模型扩展导致迁移复杂度上升 → Mitigation：字段先可空、双写兼容、分阶段启用。
- [风险] 批量接口事务失败影响用户体验 → Mitigation：返回逐行错误并支持保留本地草稿重试。
- [风险] 甘特图依赖线在大数据量下性能下降 → Mitigation：默认按阶段/时间窗过滤并限制初始渲染数量。
- [风险] 工作台复杂度过高影响上手 → Mitigation：默认展示简化视图，提供“高级模式”展开。

## Migration Plan

1. 数据库迁移：为 WBS/里程碑增加结构化字段与索引，保持旧字段可用。
2. 后端扩展：新增批量写入与计划校验接口，保留旧接口。
3. 前端接入：先在新工作台接入新接口与新模型，现有页面保持兼容。
4. 灰度验证：在部分项目上验证录入效率、校验命中率、页面性能。
5. 全量切换：默认入口指向工作台，保留旧入口一段过渡期。

回滚策略：
- 前端通过路由/特性开关回退旧入口。
- 后端保留旧接口，不依赖新字段时可继续服务。
- 必要时禁用批量接口，仅保留单条录入。

## Open Questions

- `predecessorTaskIds` 最终采用 JSON 字段还是独立关联表？
- 关键路径是否本期计算，还是先仅展示关键任务标记？
- 里程碑是否允许关联多个 WBS 任务，及其完成判定规则如何定义？
- 工作台是否需要与现有流程导航（启动/规划/执行/收尾）深度联动？

# PMP 项目管理系统 PRD（草案）

版本：v0.2（待确认）  
日期：2026-02-15

## 一、背景

当前项目管理主要依赖 Excel 模板，存在以下问题：
- 数据分散在多个工作表，跨表关联与追溯成本高。
- 权限边界弱，缺少“按项目授权”的访问控制。
- 状态评估与风险管理依赖人工口径，难以形成统一视图。
- 管理层无法快速看到项目总体健康度与风险热点。

本系统目标是将 Excel 驱动的项目管理流程系统化，形成“可录入、可追踪、可分析、可授权”的管理平台。

## 二、目标

### 2.1 业务目标

- 支撑项目从立项到执行的全过程数据管理：
  - 立项、WBS、里程碑、推进记录、状态评估、风险台账、变更申请。
- 支撑模板化导入，将既有 Excel 数据快速迁移到系统。
- 建立可落地权限模型：
  - 角色权限（ADMIN/PM/MEMBER）
  - 项目成员权限（OWNER/EDITOR/VIEWER）
- 提供项目总览看板，支撑管理决策。

### 2.2 产品目标

- 表单录入与 Excel 导入并存，兼容过渡期。
- 所有核心业务数据落库（MySQL/Prisma），不依赖内存态。
- 提供系统主数据底座（组织、字典、sys_* 实体）以支持后续扩展。

## 三、KPI（建议验收口径）

### 3.1 业务过程 KPI

- 导入成功率：模板数据清洗后可入库记录占比 >= 95%。
- 去重有效率：重复导入时，WBS/里程碑/风险/变更重复项拦截率 = 100%。
- 权限命中率：未授权项目访问与写入拦截率 = 100%。

### 3.2 使用体验 KPI

- 登录到可操作主界面耗时 <= 3 秒（本地联调环境）。
- 新增一条业务记录（非导入）平均操作步骤 <= 6 步。

### 3.3 管理价值 KPI

- 项目看板可实时给出：任务完成率、里程碑完成率、逾期任务数、开放风险数、升级风险数。
- 项目状态评估与风险热点可用于周报/例会直接引用。

## 四、用户故事

### 4.1 角色：系统管理员（ADMIN）

- 作为管理员，我希望能管理所有项目与系统配置，以保证平台可用性和治理一致性。
- 作为管理员，我希望给用户分配项目访问权限（OWNER/EDITOR/VIEWER），避免越权操作。
- 作为管理员，我希望通过系统实体管理维护 sys_* 数据，为后续权限中台能力做准备。

### 4.2 角色：项目经理（PM）

- 作为项目经理，我希望创建项目并维护全量项目过程数据，以确保项目推进可追踪。
- 作为项目经理，我希望从 Excel 一键导入历史数据，减少重复录入工作量。
- 作为项目经理，我希望查看项目总览看板，快速判断进度、里程碑与风险状态。

### 4.3 角色：项目成员（MEMBER）

- 作为项目成员，我希望在授权项目中查看项目数据，了解当前任务与风险情况。
- 作为项目成员，我希望在授权范围内提交推进记录，反馈一线执行进展。

## 五、功能需求

### 5.1 认证与用户

- 支持账号密码登录，返回 JWT。
- 支持获取当前用户信息。
- 角色体系：`ADMIN`、`PM`、`MEMBER`。

### 5.2 项目与过程管理

- 模块：
  - 项目立项卡（projects）
  - WBS 任务分解（wbs）
  - 里程碑计划（milestones）
  - 推进记录（progress-records）
  - 项目状态评估（status-assessments）
  - 风险问题台账（risks）
  - 变更申请（changes）
- 能力：列表、创建、编辑；按项目维度关联和筛选。

### 5.3 项目总览

- 提供项目级 Dashboard：
  - KPI：任务/里程碑完成率、逾期任务、开放风险、升级风险等
  - 最新状态评估
  - 风险热点（按阶段聚合）
  - 进度趋势（近 6 条记录）

### 5.4 导入与同步

- 支持 `.xlsx` 导入预览与一键入库。
- 导入前清洗模板说明行、示例行和非法行。
- 入库按项目“名称+年度”复用/新建项目。
- 具备模块级去重（WBS/里程碑/风险/变更）。

### 5.5 权限控制

- 读取权限：
  - ADMIN 可读全部项目；
  - 非 ADMIN 仅可读其已授权项目。
- 写入权限：
  - WBS/里程碑/状态评估/风险/变更：需角色 `ADMIN/PM` 且项目权限 `OWNER/EDITOR`。
  - 推进记录：角色 `ADMIN/PM/MEMBER`，但仍需项目权限 `OWNER/EDITOR`。
- 项目成员权限维护：仅项目 `OWNER`（ADMIN 具备全权）可配置。

### 5.6 系统管理

- 组织管理：组织树增删改查。
- 字典管理：项目类型、年度等字典项维护。
- 系统实体管理：对已配置 `sys_*` 实体执行通用 CRUD。

## 六、验收标准

### 6.1 功能验收

- 登录成功后，用户可按角色进入对应菜单。
- ADMIN/PM 可完成项目创建及六大过程模块录入。
- Dashboard 能返回并展示核心 KPI、风险热点与最新状态。
- Excel 预览和入库链路可用，返回创建/跳过统计。

### 6.2 权限验收

- 未登录访问受保护接口返回 401。
- 角色不符返回 403。
- 非授权项目访问返回 403。
- MEMBER 对非推进模块写入返回 403；对推进模块在无 OWNER/EDITOR 权限时仍返回 403。

### 6.3 数据规则验收

- 业务枚举字段仅接受系统定义值：
  - 阶段：启动/规划/执行/验收
  - 状态：未开始/进行中/已完成/延期
  - 信号灯：绿/黄/红
  - 是否：是/否
- 年度范围、进度百分比、日期格式满足校验要求。
- 项目名称唯一约束生效。

## 七、需求来源与可追溯性

### 7.1 来源 1：现有实现（代码）

- 后端模块路由、校验、权限中间件、Prisma 模型。
- 前端页面与菜单、模块交互流程。

### 7.2 来源 2：Excel 模板

- 字段名与列映射来源于模板工作表。
- 字段填写规范来源于：
  - 工作表正文中的“填写要求/示例”文本；
  - `xl/comments*.xml` 单元格批注（你写的“填写要求：...”）。

## 八、你问的关键问题：Excel 字段填写要求我能否读取

可以读取到，结论如下：
- 我已经验证模板中存在大量字段级填写要求，既在单元格文本中，也在批注（comments）中。
- 例如“项目目标=1句总目标+量化目标”“整体状态取三色最差值”“风险状态仅允许绿黄红”等都能读到。

但当前系统实现状态是：
- 已执行：部分枚举与基础校验、模板说明行清洗、部分去重规则。
- 未完整执行：大部分“批注级业务规则”尚未程序化（例如很多字段的强制口径、跨表一致性约束、流程状态跳转约束）。
